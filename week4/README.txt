Week 4, Project

The refactoring of code from procedural programming to object-oriented programming is a challenge. Especially when you have already begun implementing and thinking of the problem procedurally. I have essentially improved the code by coupling certain behaviour with its data by introducing OOP design. I did this through introducing a Budget class with two attributes: a pandas dataframe and the assigned base currency for the budget. All the function which previously updated the dataframe and performed calculations was now a part of the Budget class. The second class was Currency which handles loading the desired base currency and currency rates to the project and performs conversions.

I have tested the calculations on the test data that was given for this project in the TestBudget class as well as the currency conversions in the TestCurrency class. The interesting part here was testing the edge case for the currency conversions such as the method convert_to_base_from_eur. If an exchange rate is not set the method will by "default" raise a ZeroDivisionError because I have assigned the currency rates to be 0 as initial values upon object creation.

The structured could arguably be considered better because it in a more clear sense couples behaviour (methods) and data (attributes). Anyone with an OOP-background can read the code and make sense of what is going on without necessarily understanding the details. This is one of the reasons as to why the structure in OOP programming could be considered better. Another important function of using this style is to be able to test code effectively as we have done with unit tests.
